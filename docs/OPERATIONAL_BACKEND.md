# Операционный backend

Операционные данные больше не должны храниться в JSON-полях профиля или одной
записи localStorage. Production-схема использует отдельные таблицы:

- `tasks`, `task_comments`, `task_links`;
- `notifications`;
- `document_items`;
- `custody_acts`;
- `tn_issues`;
- `activity_logs`.

Все сущности связаны UUID с объектом, корпусом, этажом и дверью. RLS повторяет
область доступа пользователя к объекту и не полагается на скрытые кнопки UI.

## Автоматические просрочки

RPC `sync_overdue_door_tasks()` создаёт две задачи:

- «Передать дверь ТН» после двух дней с момента монтажа;
- «Добавить акт АОХ» после трёх дней с момента монтажа.

`automatic_key` уникален для типа задачи, двери и даты монтажа. Повторный запуск
не создаёт дубль. Когда контрольное условие закрыто, триггер переводит активную
автоматическую задачу в статус `выполнена`.

Даты `mounted_at`, `tn_accepted_at`, `custody_act_uploaded_at` и
`custody_act_closed_at` устанавливаются на стороне PostgreSQL. Клиентское время
не считается доверенным источником.

## Запуск синхронизации

На переходном этапе RPC вызывается после входа и при открытии операционных
экранов. Для production следующая ступень — плановый вызов Edge Function по cron,
чтобы контроль работал даже без открытого браузера.

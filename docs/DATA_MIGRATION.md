# Миграция данных MVP

## Цель

Иерархия `Объект -> Корпус -> Этаж -> Дверь` загружается из PostgreSQL через
`objects.getTree()`. В production дерево больше не восстанавливается из
localStorage.

## Совместимость идентификаторов

Таблицы сохраняют:

- `id`: production UUID и основной внешний ключ;
- `legacy_id`: старый строковый идентификатор MVP для повторяемого импорта.

Upsert дерева выполняется по `legacy_id`, поэтому повторный запуск не создаёт
дубликаты. После чтения приложение использует UUID.

## Порядок переноса

1. Создать компанию и реальные аккаунты.
2. Экспортировать локальное дерево объектов в JSON из контролируемого стенда.
3. Проверить ответственных: старые строковые user id нельзя переносить как UUID.
4. Войти пользователем `creator` или `company_head` в Supabase-режиме.
5. Передать проверенное дерево в `dataProvider.objects.upsertTree()`.
6. Сверить количество объектов, корпусов, этажей и дверей SQL-запросами.
7. Проверить маршруты и изменение статуса на пилотном корпусе.

Служебные уровни «Кровля» и «Паркинг» не являются этажами в БД. Паркинг хранится
флагом корпуса, кровля добавляется presentation-слоем.

## Контрольные запросы

```sql
select count(*) from objects;
select count(*) from buildings;
select count(*) from floors;
select count(*) from doors;
select floor_id, count(*) from doors group by floor_id order by floor_id;
```

Импорт сначала выполняется в staging. Production импорт разрешён только после
сверки количества и выборочной проверки карточек дверей.

# Авторизация и аккаунты

## Два режима

- `local`: демо-вход существующими mock-пользователями.
- `supabase`: реальные email/password аккаунты Supabase Auth.

В production список демо-пользователей и переключатель роли не отображаются.
Сессия хранится и обновляется Supabase SDK. После входа приложение загружает
профиль из `profiles`; профиль со статусом `disabled` немедленно выходит.

## Создание пользователя

Пользователей нельзя создавать из браузера с `service_role`. Для этого добавлена
Edge Function `invite-user`. Она:

1. проверяет JWT вызывающего пользователя;
2. разрешает управление аккаунтами только административным ролям; директор строительства может приглашать только ИТР;
3. ограничивает роли, которые может назначить руководитель;
4. создаёт одноразовую запись в `user_invitations` сроком на 7 дней;
5. отправляет стандартное приглашение Supabase Auth;
6. профиль создаётся триггером `handle_new_user` только из одобренного приглашения.

Роль и компания не читаются из редактируемых метаданных пользователя. Публичная
регистрация отключена. Первого `creator` создают операционно: запись приглашения
в БД и отправка invite через Supabase Admin API. Последующие аккаунты создаются
из раздела «Пользователи».

Deploy функции:

```bash
npx supabase functions deploy invite-user
```

## Пароль

На экране входа доступно восстановление пароля. Письмо возвращает пользователя
на `/reset-password`, где он задаёт новый пароль. Новый приглашённый пользователь
использует тот же экран, поэтому руководитель не знает и не передаёт временный
пароль.

Смена пароля требует повторного входа с текущим паролем, затем вызывается
`auth.updateUser`. Пароли никогда не сохраняются в `profiles` или localStorage
в Supabase-режиме.

## Защита ролей

Триггер `protect_profile_security_fields` не позволяет обычному пользователю
изменить себе `role`, `company_id` или `status`, даже при прямом запросе к API.
# Серверное управление пользователями

В Supabase-режиме список пользователей и их назначения загружаются из `profiles`, `object_assignments` и `building_assignments`. Изменение профиля и назначений выполняется атомарной RPC `save_profile_assignments` под RLS.

Новый аккаунт создаётся только через Edge Function `invite-user`. Она проверяет роль вызывающего пользователя, допустимую целевую роль и Origin. Директор строительства может приглашать и управлять только ИТР; руководитель компании не может создавать или изменять creator. Отключение аккаунта блокирует Auth user и профиль. Действие «Восстановить доступ» снимает обе блокировки и отправляет письмо установки нового пароля.

Для production задайте секреты функций:

```bash
npx supabase secrets set \
  APP_ALLOWED_ORIGINS="https://app.example.ru" \
  APP_PUBLIC_URL="https://app.example.ru"
```

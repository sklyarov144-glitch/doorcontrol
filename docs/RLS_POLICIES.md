# Row Level Security

RLS включён на всех таблицах ядра. Политики опираются на профиль текущего
`auth.uid()`, компанию и назначения.

## Доступ ролей

- `creator`, `company_head`: данные своей компании; только эти роли создают объекты.
- `construction_director`: назначенные объекты и их корпуса.
- `itr`: назначенные объекты/корпуса и рабочие сущности внутри них.

Для задач область ещё уже: ИТР читает и меняет только назначенные ему задачи.
Защитный trigger разрешает ему менять статус на «новая», «в работе» или
«выполнена», но запрещает подменять автора, исполнителя, срок, приоритет и
привязку к объекту. Руководители работают с задачами в пределах своей области.

Функции `can_access_object` и `can_access_building` являются единой точкой
проверки области данных. Проверки интерфейса остаются полезны для UX, но не
считаются механизмом безопасности.

Журнал `activity_logs` заполняется только защищёнными trigger-функциями. Клиентские
роли не имеют права вставлять, изменять или удалять события. Руководитель компании
видит аудит своей компании, директор строительства — аудит назначенных объектов.

## Интеграционная проверка

`supabase/tests/role_rls_matrix.test.sql` создаёт в транзакции две изолированные
компании и JWT-контексты всех четырёх ролей. Тест доказывает область чтения объектов,
корпусов и дверей, разрешённые изменения ИТР, запрет создания объекта для ИТР и
директора, финансовую изоляцию, область аудита и отзыв доступа отключённого
пользователя. Тест выполняется в database job GitHub CI после чистого применения
всех миграций.

Роль `authenticated` имеет базовые SQL-привилегии на доменные таблицы и views,
но каждая операция дополнительно ограничена RLS. Роль `anon` не имеет доступа к
бизнес-таблицам. Для `activity_logs` клиенту разрешён только `SELECT`; события
создаются исключительно защищёнными триггерами.

## Важные ограничения

- Anon key допустим во фронтенде только вместе с RLS.
- Service role key используется исключительно на доверенном сервере/в CI.
- Изменение роли и компании профиля не должно выполняться обычным клиентом.
- Перед go-live нужно повторить межтенантную проверку на staging реальными Auth JWT.

# План пилота

## Цель

Проверить на одном реальном объекте и одном корпусе, что система поддерживает ежедневную работу ИТР и даёт руководителю достоверную картину без параллельного ручного учёта.

## Объём первого пилота

- одна компания;
- один объект;
- один корпус;
- 2–3 реальных этажа;
- 2 ИТР, 1 директор строительства, 1 руководитель компании;
- реальные планы и двери, задачи, акты ОХ, замечания ТН и документы;
- период наблюдения не менее 10 рабочих дней.

## Этапы

1. Развернуть staging и применить migrations.
2. Создать компанию и пользователей через контролируемый invite flow.
3. Заполнить import template, выполнить `npm run pilot:validate -- path.json`.
4. Импортировать сначала в staging командой `npm run pilot:import -- path.json --apply`.
5. Сверить количества объектов, корпусов, этажей и дверей с исходником.
6. Выполнить UAT каждой ролью и тест RLS между двумя тестовыми компаниями.
7. Провести restore drill и нагрузочный smoke.
8. Обучить пилотную группу и запустить параллельную сверку на 3 дня.
9. После подтверждения точности сделать систему основным рабочим контуром пилота.

## Метрики успеха

- 100% дверей пилотных этажей заведены без дублей;
- ИТР обновляет рабочий статус не более чем за 30 секунд;
- руководитель получает сводку без ручного объединения файлов;
- нет утечки данных между компаниями/объектами;
- не менее 99% успешных пользовательских действий в период пилота;
- все критические дефекты закрыты до production go-live.

## Ответственные

Назначаются владелец продукта, ответственный за данные, технический ответственный и представитель ИТР. Решение о go-live принимают владелец бизнеса и технический ответственный совместно.

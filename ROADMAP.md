- [x] Supabase/PostgreSQL схема и RLS.
- [x] Supabase Auth, приглашения и управление аккаунтами.
- [x] Tenant-карточка компании и защищённое редактирование справочника компании.
- [x] Приватное файловое хранилище.
- [x] Двери, проёмы, замечания ТН и акты ОХ.

- [x] Центр проблем и управленческий дашборд.

- [x] CI с чистым database reset, lint и pgTAP/RLS-тестами.
- [x] Staging/production workflows, backup и restore drill.
- [x] Транзакционные workflow двери, акта, документа задачи и иерархии.

## В работе: пилотная готовность

- [ ] Завершить декомпозицию `LegacyApp` на независимые страницы и доменные модули.
  Экран входа, восстановление пароля, список и управление объектом/корпусами, визуализация корпуса, план этажа, маркеры, рабочая карточка двери, документы, задачи команды, ежедневный список задач, центр уведомлений, отчёты по монтажу, управление аккаунтами, фактическая матрица ролей, личный кабинет и админ-конструктор типового этажа уже вынесены в автономные модули. Компонентные тесты фиксируют права ИТР/руководителя, фактическое количество этажей без demo-дверей, генерацию и применение шаблона ко всем этажам, приватную подложку плана, переходы по уровню и двери, фильтры плана, визуальные приоритеты статусов, ролевую видимость и рабочие действия задач/уведомлений, единый backend-источник ежедневных задач, расчёт отчётов по фактическим дверям и иерархии, приглашение аккаунтов через backend, подтверждённую смену пароля, загрузку приватного аватара и единый workflow сохранения двери/акта.
- [x] Убрать оставшиеся demo-вычисления из production-пути.
  Недостижимый внутренний редактор шахматки и его localStorage-синхронизация с дверями удалены; внешние ссылки сохранены в документах. Финансовый экран и tenant-карточка работают fail-closed: при ошибке backend или нарушенной связи профиля с компанией production не подставляет сгенерированные суммы и фиктивную компанию. Локальные генераторы доступны только через явно выбранный demo provider.
- [x] Расширить component/E2E-тесты на все ключевые роли и пользовательские маршруты.
  E2E фиксирует маршрут ИТР с сохранением статуса, доступ к админ-панели, tenant/роли creator, финансовый контур руководителя и управленческий маршрут директора до назначенного корпуса. Staging и production дополнительно блокируют релиз, если любая из четырёх ролей не может войти через опубликованную форму или получает runtime-ошибку.
- [x] Провести performance-аудит больших объектов и списков.
  Каждый staging-релиз проходит HTTP smoke и 200 аутентифицированных запросов к основным доменным таблицам с ограничением p95; последний релиз подтвердил gate на hosted Supabase.
- [x] Настроить реальные staging Supabase и Vercel, включая канонический alias и release smoke.
  Каждый staging-релиз сохраняет immutable evidence с полным SHA, связанным CI run и результатами backend/auth/UI/load smoke. Ручной staging не принимается как production provenance.
- [ ] Настроить и проверить staging Sentry с тестовым событием и alert contact.
  Deployment gate и release evidence для Sentry ingestion реализованы. Текущий аудит GitHub Environment подтверждает, что `VITE_SENTRY_DSN` в staging ещё не задан; после его добавления следующий релиз отправит тестовое событие, но alert contact всё равно нужно подтвердить в Sentry.
- [x] Создать четыре staging-аккаунта и выполнить auth smoke.
- [ ] Импортировать пилотный объект и сверить контрольные количества.
  Подготовка назначений по проверенным UUID, strict preflight, атомарный импорт и reconciliation готовы. Контракт переносит статусы, фактические размеры и даты монтажа/ТН/АОХ; для закрытия пункта нужны согласованный реальный файл пилота и контрольные количества владельца данных.
- [ ] Провести UAT с руководителем и ИТР.

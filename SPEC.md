# DoorControl: краткое техническое задание

## 1. Назначение системы

DoorControl предназначен для внутреннего контроля работ по установке дверей на строительных объектах.

Система должна помогать видеть фактическое состояние дверей, проемов, замечаний и актов ответственного хранения в разрезе объектов, корпусов, секций и этажей.

MVP включает только закрытую внутреннюю часть системы. Публичный сайт-визитка компании будет добавлен позже, после запуска MVP.

## 2. Пользователи MVP

На первом этапе предполагаются следующие типы пользователей:

- Администратор системы.
- Руководитель проекта.
- Производитель работ или ответственный сотрудник на объекте.
- Сотрудник, который фиксирует статусы, замечания и документы.

Точный набор ролей и прав будет уточнен на этапе проектирования авторизации.

## 3. Основной пользовательский сценарий

1. Пользователь открывает закрытую часть системы.
2. Вводит логин и пароль.
3. После входа видит список доступных объектов.
4. Открывает объект.
5. Выбирает корпус.
6. Выбирает секцию.
7. Выбирает этаж.
8. Видит список дверей на этаже.
9. Открывает карточку двери.
10. Просматривает или обновляет статус двери, данные по проему, замечаниям и актам.

Подробный путь пользователя описан в `docs/MVP_FLOW.md`.

## 4. Базовая иерархия данных

```text
Объект
  Корпус
    Секция
      Этаж
        Дверь
```

## 5. Основные сущности MVP

### Пользователь

- ID.
- Логин.
- Пароль в защищенном виде.
- Имя.
- Роль.
- Статус активности.

### Объект

- ID.
- Название.
- Адрес.
- Статус.
- Ответственный.

### Корпус

- ID.
- Объект.
- Название или номер.

### Секция

- ID.
- Корпус.
- Название или номер.

### Этаж

- ID.
- Секция.
- Номер этажа.

### Дверь

- ID.
- Этаж.
- Номер или маркировка.
- Тип двери.
- Статус.
- Комментарий.

### Проем

- ID.
- Дверь.
- Состояние.
- Размеры, если требуются.
- Комментарий.

### Замечание

- ID.
- Дверь или проем.
- Описание.
- Статус.
- Автор.
- Дата создания.
- Дата устранения.

### Акт ответственного хранения

- ID.
- Дверь или группа дверей.
- Номер акта.
- Дата.
- Ответственный.
- Файл, фото или ссылка на файл при необходимости.

## 6. Файлы и фото

- Файлы и фото могут использоваться только для актов ответственного хранения, если это нужно по рабочему процессу.
- Для дверей, проемов и замечаний на этапе MVP достаточно статусов, комментариев и базовой истории изменений.

## 7. Статусы

Подробный список статусов дверей, проемов, замечаний и актов описан в `docs/STATUSES.md`.

Предварительный набор статусов:

- Не начато.
- Проем готов.
- Дверь доставлена.
- Установлена.
- Принята.
- Есть замечания.
- Передана на ответственное хранение.

Список статусов может быть расширен после уточнения бизнес-процесса.

## 8. Требования к MVP

- Система должна быть закрытой, без публичного сайта-визитки на первом этапе.
- Доступ только после авторизации.
- Интерфейс должен быть простым, рабочим и удобным для ежедневного использования.
- Структура объекта должна быть понятной, а навигация по ней - быстрой.
- Данные должны быть подготовлены к дальнейшему расширению отчетами и импортом.
- Сайт-визитка компании должен быть вынесен в отдельный будущий этап после MVP.

## 9. Что не входит в MVP

- Полноценный дизайн-система.
- Сложная ролевая модель.
- Интеграции с внешними системами.
- Мобильное приложение.
- Автоматическая генерация актов.
- Сложная аналитика и BI-отчеты.
- Публичный сайт-визитка компании.
